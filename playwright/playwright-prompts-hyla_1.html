Module 1

Les modules Playwright à connaître avant de prompter

Cinq briques essentielles, niveau débutant

Brique 1 / 5

@playwright/test — le moteur de test

  • C'est le seul package qu'il faut retenir pour démarrer : il inclut le test runner, les assertions et la gestion des navigateurs
  • Une commande pour tout lancer : npx playwright test
  • Chaque test tourne dans un navigateur isolé — pas d'effet de bord d'un test à l'autre
import { test, expect } from '@playwright/test'; test('la page hyla.be se charge', async ({ page }) => { await page.goto('https://www.hyla.be'); await expect(page).toHaveTitle(/Hyla/); });
Brique 2 / 5

Locators & assertions web-first

  • On ne cible jamais un élément par devinette : on le cherche comme un utilisateur le verrait — par son rôle, son texte, son label
  • getByRole('button', { name: 'Demander un test' })
  • getByText('Accepter les cookies')
  • Les assertions réessaient automatiquement jusqu'à ce que la condition soit vraie — pas besoin de sleep()
Pour le prompt : décris toujours l'élément comme tu le verrais à l'écran ("le bouton orange Demander un test"), pas comme du code.
Brique 3 / 5

Trace, capture d'écran et rapport HTML

  • Trace : enregistre chaque action, le DOM, le réseau et la console — rejouable comme une vidéo
  • Screenshot : preuve visuelle automatique en cas d'échec
  • Reporter HTML : tableau de bord cliquable, généré après chaque exécution
// playwright.config.js export default { use: { trace: 'on-first-retry', screenshot: 'on' }, reporter: 'html', };
Brique 4 / 5

Émulation d'appareils (mobile / tablette)

  • Un site comme hyla.be est majoritairement visité depuis un mobile — il faut le tester comme tel
  • Playwright simule de vrais appareils : taille d'écran, user-agent, tactile
import { devices } from '@playwright/test'; test.use({ ...devices['iPhone 13'] });
Pour le prompt : précise toujours si le test doit se faire "en version mobile" ou "en version bureau".
Brique 5 / 5

La fixture request — tester sans interface

  • Permet de vérifier qu'une page répond bien (code 200) sans ouvrir de navigateur
  • Idéal pour vérifier rapidement que des liens ne sont pas cassés
test('la page contact répond', async ({ request }) => { const res = await request.get('https://www.hyla.be/contact'); expect(res.status()).toBe(200); });
Module 2

Le terrain : découverte de hyla.be

Comprendre le site avant de le tester

Reconnaissance

Ce qu'on va tester sur hyla.be

  • Site vitrine / e-commerce du distributeur belge des purificateurs Hyla (Zaventem)
  • Éléments clés repérés : bandeau cookies, menu de navigation, bouton d'appel à l'action « Demander un test », formulaire de contact, footer avec réseaux sociaux
  • Site responsive — menu mobile à tester séparément
Bon réflexe débutant : avant d'écrire un prompt, navigue 5 minutes sur le site toi-même et note les boutons, titres et libellés exacts. Un prompt est plus fiable quand il reprend les mots réels du site.
Méthode

La structure d'un bon prompt de test

  • 1. Contexte — quelle page, quel état de départ
  • 2. Action — ce que l'utilisateur fait, dans l'ordre
  • 3. Vérification — ce qui doit être vrai après l'action
  • 4. Livrable — ce que je veux récupérer (rapport, capture, vidéo)
Exemple minimal :
"Ouvre hyla.be, accepte les cookies, vérifie que le titre de la page contient 'Hyla', donne-moi un rapport de test."
Module 3

Le pack de 10 prompts

À copier-coller tel quel, du plus simple au plus complet

⚙️ Prompt 1 · Niveau 1

Chargement de la page d'accueil

Notion travaillée : @playwright/test · assertion de base
Prompt
Écris un test Playwright qui ouvre https://www.hyla.be et vérifie que la page se charge correctement : le titre de l'onglet doit contenir "Hyla" et la page doit répondre avec un statut 200. Donne-moi le code et le résultat du test.
⚙️ Prompt 2 · Niveau 1

Bandeau de cookies

Notion travaillée : locators getByText / getByRole
Prompt
Sur https://www.hyla.be, écris un test Playwright qui vérifie que le bandeau de cookies apparaît au premier chargement, clique sur le bouton d'acceptation des cookies, puis vérifie que le bandeau disparaît bien de la page.
⚙️ Prompt 3 · Niveau 1

Menu de navigation principal

Notion travaillée : locators multiples + boucle
Prompt
Écris un test Playwright qui ouvre la page d'accueil de hyla.be, accepte les cookies, puis vérifie que tous les liens du menu de navigation principal sont visibles et cliquables. Liste-moi les liens trouvés dans le résultat du test.
⚙️ Prompt 4 · Niveau 2

Bouton « Demander un test »

Notion travaillée : interaction + vérification de navigation
Prompt
Sur hyla.be, après avoir accepté les cookies, écris un test Playwright qui clique sur le bouton "Demander un test" et vérifie qu'un formulaire ou une nouvelle page s'affiche bien à la suite de ce clic. Prends une capture d'écran après le clic.
⚙️ Prompt 5 · Niveau 2

Formulaire de demande / contact

Notion travaillée : fill, submit, messages de validation
Prompt
Écris un test Playwright qui remplit le formulaire de contact ou de demande de test sur hyla.be avec des données fictives (nom : "Test QA", email : "test.qa@example.com", téléphone : "0000000000"), soumet le formulaire, puis vérifie qu'un message de confirmation ou d'erreur de validation s'affiche. N'utilise jamais de vraies coordonnées personnelles.
⚙️ Prompt 6 · Niveau 2

Menu mobile / responsive

Notion travaillée : devices, émulation mobile
Prompt
Écris un test Playwright qui émule un iPhone 13, ouvre hyla.be, accepte les cookies, vérifie que le menu se transforme bien en menu mobile (icône hamburger), clique dessus, et vérifie que les liens de navigation deviennent visibles.
⚙️ Prompt 7 · Niveau 2

Liens du footer

Notion travaillée : locators dans une zone précise de la page
Prompt
Écris un test Playwright qui descend jusqu'au pied de page (footer) de hyla.be et vérifie que les liens vers les réseaux sociaux et les mentions légales sont bien présents et possèdent une adresse valide (attribut href non vide).
⚙️ Prompt 8 · Niveau 3

Détection de liens cassés

Notion travaillée : fixture request, codes de statut HTTP
Prompt
Écris un test Playwright qui récupère tous les liens du menu de navigation principal de hyla.be, puis vérifie pour chacun d'eux, avec la fixture "request", que la page répond avec un statut 200 et non une erreur 404 ou 500. Liste les liens en erreur s'il y en a.
⚙️ Prompt 9 · Niveau 3

Balises SEO de base

Notion travaillée : assertions sur le head HTML
Prompt
Écris un test Playwright qui vérifie que la page d'accueil de hyla.be possède bien une balise title non vide et une balise meta description non vide. Affiche leur contenu exact dans le résultat du test.
⚙️ Prompt 10 · Niveau 3

Rapport complet avec preuve visuelle

Notion travaillée : trace, screenshot, rapport HTML
Prompt
Regroupe les tests précédents (chargement de page, cookies, navigation, bouton "Demander un test") dans un seul fichier de test Playwright. Configure la trace et les captures d'écran à chaque étape, exécute les tests, puis donne-moi un rapport HTML final téléchargeable.

Module 4

Bonnes pratiques de prompting & éthique du test

Pour tester sérieusement, sans nuire au site testé

Bonne pratique 1

Toujours structurer son prompt en 4 temps

  • Contexte : quelle URL, quel état (cookies déjà acceptés ou non, mobile ou bureau)
  • Action : verbes précis et dans l'ordre — "ouvre", "clique", "remplis", "vérifie"
  • Vérification : un résultat observable, jamais une impression ("le site marche bien" n'est pas une vérification)
  • Livrable : code seul ? rapport ? capture ? — précise-le toujours
Bonne pratique 2

Les pièges classiques du débutant

  • Prompt trop vague : "teste le site" → aucune assertion possible. Précise toujours ce qui doit être vrai.
  • Oublier les cookies : sur hyla.be, le bandeau peut bloquer les clics suivants si on ne le ferme pas d'abord.
  • Utiliser de vraies données dans un formulaire de test — toujours des données fictives clairement identifiables.
  • Confondre "ça marche sur mon écran" et "le test le prouve" — un test sans assertion ne prouve rien.
Bonne pratique 3 — Important

Respecter le site qu'on teste

  • De nombreux sites, dont hyla.be, restreignent l'accès automatisé dans leur fichier robots.txt
  • Un test ponctuel pour apprendre est raisonnable — une exécution répétée ou en boucle sur un site de production ne l'est pas
  • En contexte professionnel : demander l'autorisation au propriétaire du site, ou tester sur un environnement de démo/staging dédié
  • Ne jamais soumettre de vraies données personnelles ni saturer un formulaire de contact réel
À retenir : un bon testeur QA est aussi un testeur responsable. La rigueur technique ne dispense pas du respect du site testé.
⚙️ Exercice final · Mise en pratique

Écris ton propre prompt

15 min · individuel, puis partage en groupe
  1. Choisis un élément de hyla.be non encore testé dans le pack (ex : page produit, blog, formulaire d'emploi)
  2. Rédige ton prompt en suivant la structure en 4 temps
  3. Vérifie qu'il respecte les bonnes pratiques éthiques vues précédemment
  4. Teste-le et compare ton résultat avec ton voisin
Livrable attendu : un prompt prêt à l'emploi + le code de test généré + une courte explication de ce qu'il vérifie.

@playwright/test est le seul module indispensable pour démarrer un test fonctionnel.
