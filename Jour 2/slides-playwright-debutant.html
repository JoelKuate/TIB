Module 01

Découvrir Playwright

Ce que c'est, pourquoi l'utiliser, comment l'installer

01.1 — Définition

Qu'est-ce que Playwright ?

Playwright est un outil qui permet d'écrire du code capable de piloter un vrai navigateur web à votre place : il ouvre une page, clique sur des boutons, remplit des formulaires, et vérifie ce qui s'affiche — exactement comme le ferait une personne devant son écran, mais de façon automatisée et répétable.

  • Un framework de test end-to-end créé par Microsoft, open source et gratuit
  • Il pilote un vrai navigateur (Chromium, Firefox, WebKit) — pas une simulation
  • On écrit les tests en TypeScript, JavaScript, Python, Java ou C# ; ce cours utilise TypeScript
  • Chaque test est un petit programme qui reproduit un parcours utilisateur et vérifie son résultat
Analogie simple : écrire un test Playwright, c'est un peu comme écrire une recette de cuisine très précise que quelqu'un d'autre (l'ordinateur) va suivre à la lettre, à chaque fois, sans jamais se tromper d'étape.
01.2 — Pourquoi ce choix

Pourquoi Playwright plutôt qu'un autre outil ?

Il existe plusieurs outils pour automatiser des tests de sites web (Selenium, Cypress...). Playwright a été choisi pour cette formation pour des raisons concrètes, pas juste par tendance :

  • Auto-waiting intégré : le mot "auto-waiting" signifie que Playwright attend tout seul qu'un bouton soit prêt avant de cliquer dessus — plus besoin d'écrire une pause manuelle
  • Multi-navigateurs : le même code fonctionne sur Chromium, Firefox et WebKit (le moteur de Safari)
  • Mode UI & Trace Viewer : deux outils visuels (détaillés au Module 07) qui montrent le test se dérouler et permettent de rejouer chaque étape après coup
  • Isolation des tests : chaque test démarre avec un navigateur "propre", sans données laissées par le test précédent
01.3 — Avant de commencer

Ce dont on a besoin

Playwright est un paquet qui s'installe via npm, le gestionnaire de paquets de Node.js. Avant de l'installer, il faut donc s'assurer que ces deux outils sont déjà présents sur la machine.

  • Node.js (version 18 ou plus récente) : le programme qui permet d'exécuter du JavaScript/TypeScript en dehors d'un navigateur
  • npm : installé automatiquement avec Node.js, il sert à télécharger et gérer les paquets (dont Playwright)
  • Un éditeur de code (VS Code recommandé, avec l'extension officielle Playwright)
  • Des bases en JavaScript ou TypeScript — revues en détail au Module 02
# Vérifier que tout est prêt (à taper dans un terminal)
node -v   // doit afficher une version, ex: v20.11.0
npm -v    // doit afficher une version, ex: 10.2.4
01.4 — Installation

Créer un projet Playwright

La commande ci-dessous fait tout le travail en une seule fois : elle crée les fichiers de configuration, installe le paquet Playwright, et télécharge les navigateurs nécessaires.

// Dans le dossier de votre projet
npm init playwright@latest
Comment lire cette commande
npm initdémarre un assistant d'installation interactif (pose des questions)
playwright@latestnom du paquet à installer, "@latest" = toujours la dernière version
L'assistant pose 3 questions : TypeScript ou JavaScript ? Nom du dossier de tests ? Ajouter un workflow GitHub Actions ? Répondez "Yes"/valeur par défaut si vous débutez — tout reste modifiable plus tard.
01.5 — Structure générée

Ce que l'installateur a créé

Après l'installation, quatre éléments apparaissent dans votre dossier. Voici à quoi sert chacun d'eux — vous n'aurez besoin de modifier régulièrement que le dossier tests/.

mon-projet/
├── package.json          // liste des dépendances du projet
├── playwright.config.ts  // réglages (navigateurs, dossier de tests...)
├── tests/
│   └── example.spec.ts   // un test d'exemple, prêt à lancer
└── tests-examples/       // exemples fournis par Playwright
Bon réflexe débutant : lancez tout de suite npx playwright test pour vérifier que l'exemple fourni passe au vert, avant d'écrire votre propre code. Cela confirme que l'installation a fonctionné.
Module 02

Anatomie d'un test

La syntaxe de base, expliquée pièce par pièce

02.1 — Avant le code Playwright lui-même

Trois mots-clés à comprendre d'abord

Le code Playwright utilise en permanence trois notions de JavaScript/TypeScript. Les comprendre une fois pour toutes évite de se sentir perdu sur chaque nouvelle ligne de code du reste du cours.

Le vocabulaire qui revient partout
asyncindique qu'une fonction contient des actions qui prennent du temps (charger une page, cliquer...). Toute fonction de test Playwright est async.
await"attends que ceci se termine avant de continuer". Sans await, le code passerait à la ligne suivante avant que le clic ou le chargement soit fini.
() => {}une "fonction fléchée" : un bloc de code qu'on peut passer en paramètre, exécuté plus tard par Playwright.
🧪 Mini-exemple — l'effet du mot await
// SANS await : le test essaie de vérifier le titre AVANT que la page soit chargée → échoue souvent
page.goto('https://playwright.dev');
expect(page).toHaveTitle(/Playwright/);

// AVEC await : le test attend la fin du chargement avant de continuer → fiable
await page.goto('https://playwright.dev');
await expect(page).toHaveTitle(/Playwright/);
C'est l'erreur la plus fréquente chez les débutants : oublier await rend un test instable de façon imprévisible.
02.2 — Les deux briques de base

test() et expect()

Chaque fichier de test importe deux fonctions fournies par Playwright : test pour déclarer un cas de test, et expect pour vérifier un résultat. C'est le squelette que l'on retrouve dans 100% des fichiers de test.

import { test, expect } from '@playwright/test';

test('mon premier test', async ({ page }) => {
  // "page" représente l'onglet du navigateur piloté par le test
});
Décorticage de la ligne test(...)
'mon premier test'le nom du test, affiché dans les résultats — choisissez un nom qui décrit CE QUI est vérifié
async ({ page })la fonction à exécuter ; Playwright lui fournit automatiquement l'objet page (une "fixture")
{ page }syntaxe de "déstructuration" : on récupère uniquement la propriété page de l'objet fourni par Playwright
🧪 Mini-exemple — deux tests dans un même fichier
test('le titre est correct', async ({ page }) => { /* ... */ });
test('le bouton est visible', async ({ page }) => { /* ... */ });
Un fichier peut contenir plusieurs test() : chacun reçoit sa PROPRE page, totalement indépendante des autres.
02.3 — Naviguer

page.goto()

C'est presque toujours la première ligne d'un test : on demande au navigateur d'ouvrir une adresse web, comme si on la tapait dans la barre d'adresse.

test('ouvrir la page d\'accueil', async ({ page }) => {   // on déclare le test
  await page.goto('https://www.saucedemo.com');   // on ouvre l'adresse et on ATTEND la fin du chargement
});
Décorticage
pagel'objet qui représente l'onglet du navigateur (fourni automatiquement, voir slide précédente)
.goto('url')une "méthode" du navigateur : elle demande le chargement d'une adresse
awaitobligatoire ici : le chargement d'une page prend du temps, on attend qu'il se termine
🧪 Mini-exemple — avec une baseURL configurée
// Si playwright.config.ts définit baseURL: 'https://www.saucedemo.com'
await page.goto('/');              // équivaut à https://www.saucedemo.com/
await page.goto('/checkout.html');  // équivaut à https://www.saucedemo.com/checkout.html
Utile pour ne pas répéter l'adresse complète du site dans chaque test (voir Module 07 pour la configuration).
02.4 — Premier test complet

Assembler les pièces : un test complet

Voici les trois notions vues jusqu'ici (import, test(), page.goto()) combinées avec une première vérification via expect(). C'est le plus petit test "utile" qu'on puisse écrire.

import { test, expect } from '@playwright/test';

test('la page a le bon titre', async ({ page }) => {
  await page.goto('https://playwright.dev/');          // 1. on navigue

  await expect(page).toHaveTitle(/Playwright/);       // 2. on vérifie
});
Décorticage de la ligne 2
expect(page)on déclare qu'on va vérifier quelque chose à propos de l'objet page
.toHaveTitle(...)l'assertion précise : "le titre de l'onglet doit correspondre à..."
/Playwright/une expression régulière : elle matche tout titre CONTENANT le mot "Playwright" (plus souple qu'un texte exact)
Le schéma à retenir : naviguer → interagir → vérifier. Presque tous les tests Playwright suivent ces trois étapes, dans cet ordre. Les Modules 03 à 05 détaillent chacune de ces trois étapes.
Module 03

Les sélecteurs (locators)

Comment désigner un élément précis sur la page

03.1 — Le sujet le plus important du cours

Pourquoi bien choisir son sélecteur ?

Avant de cliquer sur un bouton ou de vérifier un texte, il faut d'abord dire à Playwright QUEL élément de la page on désigne. C'est le rôle d'un locator — littéralement, un objet qui "localise" un ou plusieurs éléments.

  • Un locator ne "cherche" pas immédiatement l'élément : il garde juste la façon de le retrouver, à chaque action
  • Un mauvais sélecteur rend le test fragile : il casse au moindre changement visuel du site
  • Playwright recommande un ordre de préférence, du plus robuste au plus fragile :
✓ À privilégier
  • getByRole()
  • getByLabel()
  • getByTestId()
✗ À éviter
  • Sélecteurs CSS trop précis (classes générées automatiquement)
  • XPath complexe
03.2 — Le sélecteur recommandé n°1

getByRole()

Chaque élément d'une page a un "rôle" d'accessibilité (bouton, lien, titre, case à cocher...), utilisé notamment par les lecteurs d'écran pour les personnes malvoyantes. getByRole() cible un élément par ce rôle, combiné à son texte visible.

page.getByRole('button', { name: 'Se connecter' });        // cible le bouton dont le libellé est "Se connecter"
page.getByRole('link',   { name: 'Contact', exact: true }); // cible le lien "Contact" — texte EXACT exigé
page.getByRole('heading',{ name: 'Bienvenue', level: 1 });   // cible le titre <h1> "Bienvenue"
Décorticage
'button'le rôle recherché (aussi : 'link', 'checkbox', 'heading', 'textbox'...)
{ name: '...' }le texte visible associé à cet élément — sans lui, TOUS les boutons de la page seraient ciblés
exact: trueoption pour exiger une correspondance exacte plutôt qu'une simple sous-chaîne
🧪 Mini-exemple — cibler une case à cocher
// cible la case à cocher associée au texte "Se souvenir de moi"
page.getByRole('checkbox', { name: 'Se souvenir de moi' });
Même logique, quel que soit le type d'élément : rôle + nom visible.
Pourquoi c'est le meilleur choix : c'est ce qu'un lecteur d'écran verrait aussi — donc proche de ce qu'un vrai utilisateur perçoit, et peu sensible aux changements de design (classes CSS, structure HTML).
03.3 — Cibler par contenu visible

getByText, getByLabel, getByPlaceholder

Trois variantes plus spécialisées, utiles quand le rôle d'accessibilité n'est pas assez précis ou pas disponible.

page.getByText('Bienvenue');                    // cible un texte affiché n'importe où
page.getByLabel('Adresse e-mail');              // cible le CHAMP associé à ce label
page.getByPlaceholder('Rechercher...');         // cible via le texte indicatif d'un input
Trois usages différents à ne pas confondre
getByTextrenvoie l'élément qui AFFICHE ce texte (un titre, un paragraphe, un lien...)
getByLabelrenvoie le CHAMP DE SAISIE lié à ce label — pas le texte du label lui-même
getByPlaceholderutile quand un champ n'a pas de label visible, seulement un texte grisé à l'intérieur
🧪 Mini-exemple — remplir un champ trouvé par son label
// on trouve le champ via son étiquette visible, puis on le remplit
await page.getByLabel('Adresse e-mail').fill('test@exemple.be');
getByLabel() est idéal pour les formulaires : il fonctionne même si le champ n'a pas d'id ni de data-testid.
Par défaut, la recherche est insensible à la casse et par sous-chaîne. Ajoutez { exact: true } pour exiger une correspondance exacte.
03.4 — Le sélecteur le plus stable

getByTestId()

Certains sites ajoutent volontairement un attribut dédié aux tests dans leur HTML (invisible pour l'utilisateur, mais lisible par Playwright). C'est le sélecteur le plus fiable, car il ne dépend ni du texte affiché, ni du design.

// Dans le HTML : <button data-testid="login-button">Login</button>
page.getByTestId('login-button');
Décorticage
data-testidl'attribut HTML standard que Playwright recherche par défaut
'login-button'la valeur exacte de cet attribut sur l'élément recherché
🧪 Mini-exemple — adapter à un attribut différent
// playwright.config.ts — certains sites (ex: saucedemo.com) utilisent "data-test" au lieu de "data-testid"
use: {
  testIdAttribute: 'data-test',
}
// Le code de test ne change pas : page.getByTestId('login-button') fonctionne toujours
Un seul réglage dans la configuration suffit à adapter getByTestId() à n'importe quel attribut personnalisé.
03.5 — Le sélecteur généraliste

locator() — sélecteurs CSS

Quand rien d'autre n'est disponible, locator() accepte n'importe quel sélecteur CSS classique — la même syntaxe qu'en feuille de style. C'est le sélecteur le plus puissant, mais aussi le plus fragile.

page.locator('.product-card');          // par classe CSS (le point .)
page.locator('#login-form');            // par id (le dièse #)
page.locator('button[type="submit"]');  // par attribut, entre crochets
page.locator('ul.products li.product'); // combiné : élément DANS un autre élément
Rappel de syntaxe CSS
.classecible tous les éléments qui ont cette classe CSS
#identifiantcible l'UNIQUE élément qui a cet id (un id doit être unique sur une page)
a b(espace entre deux sélecteurs) cible "b" à l'intérieur de "a", même profondément imbriqué
Utile quand la page n'a ni rôle exploitable ni data-testid — c'est l'échappatoire, pas le premier réflexe (voir slide 03.1).
03.6 — Affiner sa sélection

Combiner et filtrer un locator

Il arrive qu'un sélecteur trouve PLUSIEURS éléments (ex : tous les produits d'une liste). Playwright fournit des outils pour choisir précisément lequel utiliser.

page.locator('li.product').first();             // le premier élément trouvé
page.locator('li.product').nth(2);               // le 3e élément (index 0)
page.locator('li.product').filter({ hasText: 'Sac à dos' });
page.locator('.card').locator('button');          // locator imbriqué dans un autre
Décorticage
.first() / .last()choisit le premier ou le dernier élément d'une liste de résultats
.nth(2)choisit l'élément à la position 2 — attention, le comptage commence à 0 !
.filter({hasText})garde uniquement les éléments qui contiennent ce texte quelque part à l'intérieur
Piège fréquent : un locator qui matche plusieurs éléments fait échouer une action avec une "erreur de mode strict" (strict mode violation). C'est volontaire : Playwright refuse de deviner lequel vous vouliez. Utilisez .first(), .nth() ou un filtre pour lever l'ambiguïté.
Module 04

Les actions utilisateur

Cliquer, saisir, cocher — simuler un vrai utilisateur

04.1 — Interactions à la souris

click(), dblclick(), hover()

Une fois un élément localisé (Module 03), on peut agir dessus. Ces trois méthodes reproduisent les gestes de souris les plus courants.

await page.getByRole('button', { name: 'Ajouter au panier' }).click();   // clic simple
await page.locator('.item').dblclick();                                 // double-clic
await page.getByRole('link', { name: 'Shop' }).hover();                 // survol (sans cliquer)
Décorticage
.click()simule un clic gauche, à l'endroit exact où l'utilisateur cliquerait
.dblclick()deux clics rapprochés (rare, utile pour certains widgets)
.hover()place le curseur SANS cliquer — utile pour révéler un menu déroulant
Avant de cliquer, Playwright attend automatiquement que l'élément soit visible, stable et non recouvert par un autre élément (auto-waiting — détaillé au Module 06).
04.2 — Saisie de texte

fill() et press()

Pour remplir un champ de formulaire, deux méthodes complémentaires : l'une écrit le texte d'un coup, l'autre simule l'appui d'une touche précise du clavier.

await page.getByLabel('Identifiant').fill('standard_user');   // remplit le champ (efface l'ancien contenu)
await page.getByLabel('Mot de passe').fill('secret_sauce');   // idem pour le mot de passe
await page.getByLabel('Mot de passe').press('Enter');          // simule l'appui sur la touche Entrée
Décorticage
.fill('texte')efface le contenu actuel du champ et le remplace entièrement par "texte", en une seule opération rapide
.press('Enter')simule l'appui d'UNE touche précise (aussi : 'Tab', 'Escape', 'ArrowDown'...)
🧪 Mini-exemple — pourquoi éviter type()
// type() existe mais est déconseillé : frappe touche par touche, donc plus lent
await page.getByLabel('Recherche').type('chaussures'); // ⚠️ lent
await page.getByLabel('Recherche').fill('chaussures');  // ✓ rapide et fiable
fill() est quasiment toujours le bon choix, sauf si vous testez spécifiquement un comportement JavaScript qui réagit à chaque touche tapée.
04.3 — Cases à cocher et listes

check(), uncheck(), selectOption()

Trois méthodes dédiées aux éléments de formulaire à choix : cases à cocher et menus déroulants.

await page.getByLabel('Se souvenir de moi').check();               // coche la case
await page.getByLabel('Newsletter').uncheck();                     // décoche la case
await page.getByLabel('Trier par').selectOption('Prix croissant'); // choisit par texte affiché
await page.getByLabel('Pays').selectOption({ value: 'BE' });      // choisit par valeur interne
Décorticage
.check()coche la case si elle ne l'est pas déjà (ne fait rien si déjà cochée)
.uncheck()décoche la case si elle est cochée
.selectOption(...)choisit une option dans un menu déroulant — par son texte visible OU par sa valeur technique value
Utilisez .check() plutôt que .click() sur une case à cocher : l'intention est plus claire dans le code, et Playwright vérifie lui-même que la case est bien cochable.
04.4 — Actions complémentaires

Upload, glisser-déposer, défilement

Ces actions sont moins fréquentes que click/fill mais reviennent régulièrement dès qu'on teste des formulaires avancés ou des interfaces interactives.

await page.getByLabel('Fichier').setInputFiles('photo.jpg');       // envoie un fichier dans un champ upload
await page.locator('.carte').dragTo(page.locator('.corbeille'));   // glisser-déposer d'un élément vers un autre
await page.locator('footer').scrollIntoViewIfNeeded();             // fait défiler jusqu'à rendre l'élément visible
Décorticage
.setInputFiles(...)sélectionne un fichier local pour un champ de type "upload", sans ouvrir de fenêtre système
.dragTo(cible)glisse le premier élément jusqu'à la position du second (glisser-déposer)
.scrollIntoViewIfNeeded()fait défiler la page pour amener l'élément dans la zone visible, seulement si nécessaire
Module 05

Les assertions

expect() — vérifier que ce qui s'affiche est bien ce qu'on attend

05.1 — Assertions de visibilité et d'état

toBeVisible(), toBeEnabled()...

Une assertion Playwright suit toujours le même schéma : expect(élément).assertion(). Ce groupe de méthodes vérifie l'ÉTAT d'un élément — sans regarder son contenu texte.

await expect(page.getByText('Bienvenue')).toBeVisible();   // l'élément doit être visible à l'écran
await expect(page.locator('.bandeau')).toBeHidden();       // l'élément doit être caché (ou absent)
await expect(page.getByRole('button')).toBeEnabled();      // le bouton doit être cliquable (pas grisé)
await expect(page.getByLabel('CGV')).toBeChecked();        // la case doit être cochée
Décorticage
toBeVisible()l'élément existe ET est affiché à l'écran (pas caché par du CSS)
toBeHidden()l'inverse : l'élément est absent ou masqué
toBeEnabled()l'élément peut être cliqué/rempli (pas grisé/désactivé)
Toutes les assertions expect() en Playwright sont auto-attendantes : si l'élément met 500ms à apparaître, le test ne casse pas — Playwright réessaie automatiquement jusqu'à un délai maximum (5 secondes par défaut).
05.2 — Assertions de texte et de valeur

toHaveText(), toHaveValue()

Ce groupe vérifie le CONTENU d'un élément : le texte affiché, ou la valeur saisie dans un champ.

await expect(page.locator('h1')).toHaveText('Produits');         // texte exact
await expect(page.locator('.prix')).toContainText('€');          // sous-chaîne
await expect(page.getByLabel('Quantité')).toHaveValue('2');         // champ de formulaire
Décorticage — bien choisir entre les deux premières
toHaveText()exige une correspondance EXACTE du texte (attention aux espaces, majuscules)
toContainText()vérifie juste que le texte CONTIENT cette sous-chaîne — plus tolérant, souvent préférable
toHaveValue()spécifique aux champs de saisie (input, textarea) — lit ce qui a été TAPÉ, pas ce qui est affiché autour
05.3 — Assertions de page et de liste

toHaveURL(), toHaveCount()

Ces assertions ne portent pas sur un élément précis, mais sur la PAGE entière (son adresse, son titre) ou sur le NOMBRE d'éléments trouvés par un locator.

await expect(page).toHaveURL(/\/panier\//);              // l'URL doit contenir "/panier/"
await expect(page).toHaveTitle(/Boutique/i);             // le titre contient "Boutique" (i = ignore la casse)
await expect(page.locator('li.product')).toHaveCount(6); // il doit y avoir exactement 6 produits listés
Décorticage
expect(page)ici on vérifie la page elle-même, pas un élément — remarquez l'absence de .locator()
/panier/une expression régulière : les barres obliques / / délimitent un motif de recherche souple
/Boutique/ile "i" final = insensible à la casse (Boutique = boutique = BOUTIQUE)
Bonne pratique : préférez une expression régulière (/panier/) à une chaîne de caractères exacte pour l'URL — plus tolérant aux paramètres de requête (?id=42) sans changer le sens du test.
05.4 — Cas avancé

expect.poll() — attendre une valeur asynchrone

Certaines valeurs (un compteur mis à jour par un appel réseau séparé, par exemple) ne sont pas couvertes par les assertions habituelles. expect.poll() relit une valeur en boucle jusqu'à ce qu'elle corresponde.

// expect.poll : ré-évalue une valeur en boucle jusqu'à ce qu'elle satisfasse la condition
await expect
  .poll(async () => Number(await badge.textContent()), { timeout: 10000 })  // relit le compteur pendant 10 s max
  .toBeGreaterThan(0);   // la valeur lue doit finir par être > 0
Décorticage
.poll(fn, opts)appelle la fonction fn plusieurs fois de suite, jusqu'à ce que le résultat passe l'assertion suivante
timeout: 10000délai maximum en millisecondes (10000 = 10 secondes) avant d'abandonner
.toBeGreaterThan(0)la condition attendue, appliquée à chaque nouvelle valeur lue
Utile quand un badge de panier est rafraîchi par un appel AJAX séparé de l'action qui l'a déclenché — un cas fréquent sur les vrais sites marchands.
Module 06

Attente & stabilité

Pourquoi Playwright a (presque) toujours raison d'attendre

06.1 — Le principe le plus important

L'auto-waiting expliqué

Une page web ne s'affiche jamais instantanément : images qui chargent, animations, contenu ajouté par JavaScript. Playwright gère ce délai tout seul, sans qu'on ait à l'écrire.

Avant chaque action (click, fill...), Playwright vérifie que l'élément est :

  • Attaché au DOM
  • Visible
  • Stable (pas en train d'animer)
  • Non désactivé
  • Non recouvert par un autre élément
  • Prêt à recevoir l'action
Ces vérifications se font automatiquement, en arrière-plan, sans code supplémentaire — c'est ce qui rend les tests Playwright naturellement plus stables que ceux d'outils plus anciens (qui exigeaient d'écrire ces attentes à la main).
06.2 — Attentes explicites

waitForURL(), waitForLoadState()

L'auto-waiting couvre les actions sur un élément précis. Pour attendre un événement plus global (une redirection, la fin des requêtes réseau), on utilise ces méthodes explicites.

await page.getByRole('button', { name: 'Connexion' }).click();
await page.waitForURL('**/catalogue.html');      // attend la redirection

await page.waitForLoadState('networkidle'); // attend la fin des requêtes réseau
Décorticage
waitForURL('**/x')les deux étoiles ** signifient "n'importe quel début d'adresse" — utile pour matcher un domaine dynamique
waitForLoadState'networkidle' = plus aucune requête réseau active depuis 500ms (page "au repos")
À utiliser quand une action déclenche une navigation ou un chargement dont la fin n'est pas capturée par une simple assertion sur un élément.
06.3 — L'anti-pattern à connaître

Le piège de waitForTimeout()

Il existe une méthode qui met le test en pause pendant un temps fixe — mais elle est presque toujours une mauvaise idée. Comparons deux façons d'obtenir le même résultat.

✗ À éviter
await page.click('button');          // on clique
await page.waitForTimeout(3000);     // ❌ attente FIXE de 3 s : trop long si rapide, trop court si lent
await expect(banner).toBeVisible();  // la vérification peut échouer quand même
✓ À privilégier
await page.click('button');            // on clique
await expect(banner)                   // ✅ attente INTELLIGENTE : Playwright ré-essaie
  .toBeVisible({ timeout: 10000 });   // jusqu'à 10 s max, puis passe dès que c'est visible
Pourquoi : une pause fixe (waitForTimeout) est soit trop courte (le test reste instable si le site répond parfois plus lentement), soit trop longue (le test devient lent inutilement à chaque exécution). Une assertion avec attente intégrée s'adapte, elle, au temps réel de réponse — elle continue dès que la condition est vraie, sans attendre inutilement.

Lancer & déboguer ses tests

Les commandes du quotidien

07.1 — Le strict nécessaire

Commandes CLI essentielles

npx est l'outil qui permet d'exécuter un programme installé dans le projet (ici, Playwright) sans avoir à l'installer globalement sur la machine.

npx playwright test                        # lance toute la suite de tests
npx playwright test mon-fichier.spec.ts    # un seul fichier
npx playwright test -g "connexion"        # filtre par nom de test (-g = grep)
npx playwright test --headed               # navigateur VISIBLE pendant l'exécution
npx playwright test --project=chromium     # un seul navigateur, plus rapide
npx playwright test --debug                # pas à pas, avec inspecteur
Décorticage
-g "texte"ne lance que les tests dont le NOM contient ce texte (pratique pour tester un seul cas)
--headedpar défaut Playwright tourne "headless" (sans fenêtre visible) ; ce drapeau affiche le navigateur
07.2 — Les meilleurs amis du débutant

Mode UI & Trace Viewer

Deux outils visuels indispensables pour comprendre ce que fait réellement un test, sans deviner à partir du seul texte d'erreur.

npx playwright test --ui              # mode interactif
npx playwright show-report            # rapport HTML après exécution
npx playwright show-trace trace.zip   # rejoue un test étape par étape
Ce que fait chaque outil
--uiliste tous les tests, permet d'en lancer un seul en direct, et d'inspecter le DOM à chaque étape
show-reportouvre un tableau récapitulatif : durée, statut, capture d'écran de chaque test
show-tracerejoue littéralement l'enregistrement d'un test échoué, action par action, avec le DOM de chaque instant
Réflexe à prendre : face à un test qui échoue, ouvrez le Trace Viewer AVANT de modifier le code — 90% du temps, la cause est visible immédiatement.
07.3 — Générer du code automatiquement

playwright codegen

Plutôt que d'écrire un sélecteur à la main, on peut demander à Playwright de l'observer pendant qu'on clique soi-même dans le navigateur.

# ouvre un navigateur + un enregistreur : vos clics sont traduits en code Playwright
npx playwright codegen https://www.saucedemo.com
Comment ça marche
codegen <url>ouvre DEUX fenêtres : le navigateur cible, et une fenêtre "Playwright Inspector" qui affiche le code au fur et à mesure
chaque clicgénère automatiquement la ligne page.getByRole(...).click() correspondante
Excellent pour découvrir les sélecteurs d'un site inconnu — à utiliser comme point de départ, puis à nettoyer (préférer getByRole à ce que codegen propose parfois de plus fragile).
07.4 — La configuration du projet

playwright.config.ts — l'essentiel

Ce fichier centralise les réglages appliqués à TOUS les tests du projet, pour éviter de les répéter dans chaque fichier.

export default defineConfig({                    // la configuration globale du projet
  testDir: './tests',                            // dossier où Playwright cherche les tests
  use: {                                         // réglages appliqués à TOUS les tests
    baseURL: 'https://www.saucedemo.com',        // adresse de base → page.goto('/') suffit ensuite
    trace: 'on-first-retry',                     // enregistre une trace de débogage si un test rejoue
    screenshot: 'only-on-failure',               // capture d'écran uniquement en cas d'échec
  },
  projects: [                                    // liste des navigateurs à tester
    { name: 'chromium', use: { ...devices['Desktop Chrome'] } },  // ici : Chrome de bureau
  ],
});
Décorticage des options clés
testDirle dossier où Playwright va chercher les fichiers *.spec.ts
baseURLpermet d'écrire page.goto('/') au lieu de l'adresse complète (voir slide 02.3)
screenshot'only-on-failure' = une capture d'écran est prise automatiquement, uniquement si le test échoue

Exercices sur un site exemple

saucedemo.com — site officiel de démo, conçu pour l'automatisation

⚙️ Exercice pratique 1

Tester une connexion réussie

15 min · Individuel · Site : https://www.saucedemo.com
  1. Créez un fichier connexion.spec.ts dans tests/
  2. Naviguez vers https://www.saucedemo.com
  3. Remplissez les champs avec standard_user / secret_sauce
  4. Cliquez sur le bouton de connexion
  5. Vérifiez que l'URL contient inventory.html
test('connexion réussie', async ({ page }) => {
  await page.goto('https://www.saucedemo.com');               // 1. naviguer vers le site
  await page.getByTestId('username').fill('standard_user');  // 2. remplir l'identifiant
  await page.getByTestId('password').fill('secret_sauce');   // 3. remplir le mot de passe
  await page.getByTestId('login-button').click();            // 4. cliquer sur "Login"

  await expect(page).toHaveURL(/inventory\.html/);           // 5. vérifier qu'on a bien atterri sur le catalogue
});
⚙️ Exercice pratique 2

Ajouter un produit au panier

15 min · Individuel · À partir du test précédent
  1. Reprenez la connexion de l'exercice 1
  2. Cliquez sur "Add to cart" pour le produit "Sauce Labs Backpack"
  3. Vérifiez que le badge du panier affiche "1"
  4. Vérifiez que le bouton devient "Remove"
// Action : ajouter le sac à dos au panier
await page.getByTestId('add-to-cart-sauce-labs-backpack').click();

// Vérifications : le badge du panier affiche "1" et le bouton devient "Remove"
await expect(page.getByTestId('shopping-cart-badge')).toHaveText('1');
await expect(page.getByTestId('remove-sauce-labs-backpack')).toBeVisible();
Livrable attendu : un test qui passe au vert avec npx playwright test --headed --project=chromium
⚙️ Exercice pratique 3 — bonus

Un cas négatif : mauvais mot de passe

10 min · Individuel · Tester l'échec, pas seulement le succès
  1. Remplissez le formulaire avec un identifiant valide et un mauvais mot de passe
  2. Cliquez sur connexion
  3. Vérifiez qu'un message d'erreur s'affiche
  4. Vérifiez qu'on reste sur la page de connexion (pas de redirection)
await expect(page.locator('[data-test="error"]')).toBeVisible();  // le message d'erreur doit s'afficher
await expect(page).not.toHaveURL(/inventory\.html/);              // .not = on vérifie qu'on n'est PAS entré
Pourquoi cet exercice compte : un testeur qui ne vérifie que les cas de succès laisse passer la moitié des bugs. Tester l'échec attendu est aussi important que tester la réussite.
Avant de conclure

Bonnes pratiques à retenir

  • Préférez getByRole/getByLabel/getByTestId aux sélecteurs CSS fragiles
  • Ne jamais utiliser waitForTimeout() comme solution par défaut
  • Un test = un comportement vérifié, avec un nom explicite
  • Testez aussi les cas négatifs (champs vides, mauvaises données)
  • Utilisez le Trace Viewer avant de chercher ailleurs quand un test échoue
  • Sur un vrai site en production (pas un site de démo) : limitez la fréquence des exécutions
