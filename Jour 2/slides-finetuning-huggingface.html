Module 01

Comprendre le fine-tuning

Ce que c'est, quand l'utiliser, quand s'en passer

01.1 — Définition

Qu'est-ce que le fine-tuning ?

Un modèle de langage pré-entraîné (comme Llama, Mistral ou BERT) a déjà appris la structure générale du langage sur d'énormes volumes de texte. Le fine-tuning consiste à poursuivre son entraînement sur un jeu de données plus petit et spécialisé, pour qu'il adapte son comportement à une tâche ou un domaine précis.

  • On ne repart jamais de zéro : on part d'un modèle déjà compétent en langage général
  • On ajuste ses poids internes (les paramètres du réseau de neurones) à partir de nouveaux exemples
  • Le résultat est un modèle qui reproduit mieux un style, un format, ou un vocabulaire spécifique
Analogie : c'est l'équivalent d'un médecin généraliste qui suit une spécialisation en cardiologie — il garde toutes ses connaissances de base, mais devient nettement plus pointu sur un domaine précis.
01.2 — Situer le fine-tuning

Trois leviers pour personnaliser un LLM

Le fine-tuning n'est qu'une des trois façons d'adapter un modèle à un besoin métier. Elles ne s'excluent pas : on les combine souvent.

LevierCe qu'on changeCoûtIdéal pour
Prompt engineeringRien dans le modèle — juste la question poséeQuasi nulPrototypage rapide, cas simples
RAGRien dans le modèle — on lui fournit des documents au moment de la questionFaible à modéréConnaissances factuelles à jour
Fine-tuningLes poids internes du modèle lui-mêmeModéré à élevéStyle, format, vocabulaire, tâche répétitive
Règle de décision courante : commencer par le prompt engineering, ajouter du RAG si le modèle manque de connaissances, et ne fine-tuner que si le comportement lui-même (pas la connaissance) doit changer durablement.
01.3 — Cas d'usage réels

Quand fine-tuner (et quand s'en passer)

✓ Bons cas d'usage
  • Adopter un ton de marque constant
  • Produire un format de sortie précis (JSON structuré, code)
  • Maîtriser un vocabulaire technique ou une langue peu représentée
  • Réduire un prompt système trop long et répété
✗ À éviter
  • Ajouter des faits qui changent souvent (préférer le RAG)
  • Corriger un besoin qu'un meilleur prompt résoudrait déjà
  • Sans données de qualité en quantité suffisante
Piège fréquent : fine-tuner pour "apprendre des faits" au modèle est souvent une mauvaise idée — le modèle les mémorise mal et ils deviennent vite obsolètes. Le RAG est presque toujours préférable pour de la connaissance factuelle.
01.4 — Les approches techniques

Full fine-tuning, PEFT, instruction tuning

Toutes les approches ne modifient pas le modèle de la même façon. Ce cours se concentre sur PEFT (et LoRA en particulier), l'approche la plus accessible aujourd'hui.

  • Full fine-tuning : on réentraîne TOUS les paramètres du modèle — très coûteux en mémoire GPU, réservé aux gros moyens
  • PEFT (Parameter-Efficient Fine-Tuning) : on n'entraîne qu'une petite fraction des paramètres (souvent <1%) — l'approche LoRA/QLoRA, détaillée au Module 05
  • Instruction tuning : un fine-tuning particulier où les données sont des paires "instruction → réponse", pour apprendre à un modèle à mieux suivre des consignes
Dans l'immense majorité des projets réels aujourd'hui, on utilise PEFT (LoRA) plutôt que le full fine-tuning : le gain de qualité est proche, pour une fraction du coût en calcul et en mémoire.
Module 02

L'écosystème Hugging Face

Le Hub et les bibliothèques qu'on va utiliser tout au long du cours

02.1 — Le point de départ

Le Hub Hugging Face

Hugging Face Hub (huggingface.co) est une plateforme communautaire qui héberge des centaines de milliers de modèles, datasets et démonstrations, la plupart réutilisables librement.

  • Models : modèles pré-entraînés prêts à l'emploi ou à fine-tuner (BERT, Llama, Mistral, Whisper...)
  • Datasets : jeux de données prêts à charger en une ligne de code
  • Spaces : démonstrations interactives hébergées (souvent avec Gradio ou Streamlit)
  • Chaque ressource a un identifiant unique au format organisation/nom, ex : meta-llama/Llama-3.1-8B
Réflexe à prendre : avant de fine-tuner quoi que ce soit, cherchez sur le Hub si un modèle déjà fine-tuné pour un besoin proche du vôtre n'existe pas — cela évite parfois tout le travail.
02.2 — La bibliothèque centrale

transformers

La bibliothèque Python qui permet de charger, utiliser et entraîner la quasi-totalité des modèles du Hub avec une API unifiée — quel que soit le modèle, le code pour le charger se ressemble toujours.

from transformers import AutoTokenizer, AutoModelForSequenceClassification

model_name = "distilbert-base-uncased"
tokenizer = AutoTokenizer.from_pretrained(model_name)
model = AutoModelForSequenceClassification.from_pretrained(model_name, num_labels=2)
Décorticage
AutoTokenizerdécoupe le texte en "tokens" (unités que le modèle comprend) — le bon tokenizer est choisi automatiquement selon le modèle
AutoModelFor...charge l'architecture adaptée à la tâche (classification, génération de texte...) ; le préfixe "Auto" évite de connaître la classe exacte
.from_pretrained()télécharge (ou réutilise en cache) les poids déjà entraînés depuis le Hub, à partir de l'identifiant du modèle
02.3 — Charger des données

datasets

La bibliothèque datasets permet de charger un jeu de données du Hub (ou local) en une seule ligne, avec un format standardisé quel que soit la source.

from datasets import load_dataset

dataset = load_dataset("imdb")
print(dataset)
Décorticage
load_dataset("imdb")télécharge le dataset "imdb" (critiques de films) depuis le Hub — même principe que from_pretrained() pour les modèles
datasetun objet qui contient généralement des sous-ensembles nommés ("train", "test") accessibles par clé, comme un dictionnaire
🧪 Mini-exemple — charger son propre fichier
dataset = load_dataset("json", data_files="mon_dataset.jsonl")
On n'est pas limité aux datasets publics : le même outil charge un fichier JSON Lines local avec la même API.
02.4 — Fine-tuning léger

peft — LoRA et QLoRA

La bibliothèque peft (Parameter-Efficient Fine-Tuning) de Hugging Face implémente les techniques qui permettent de fine-tuner un modèle en n'entraînant qu'une infime fraction de ses paramètres. Détaillée en profondeur au Module 05.

  • LoraConfig : configure la méthode LoRA (Low-Rank Adaptation)
  • get_peft_model() : "enveloppe" un modèle existant pour n'entraîner que les nouvelles petites matrices ajoutées
  • Fonctionne avec presque tous les modèles de transformers, sans les modifier
Sans peft, fine-tuner un modèle de plusieurs milliards de paramètres nécessite plusieurs GPU professionnels. Avec LoRA, c'est souvent possible sur une seule carte grand public.
02.5 — Pour les grands modèles de langage

trl et SFTTrainer

Pour le fine-tuning par instructions de LLM génératifs (type chatbot), Hugging Face propose trl (Transformer Reinforcement Learning), avec un entraîneur spécialisé qui simplifie encore le code du Module 04.

from trl import SFTTrainer, SFTConfig

trainer = SFTTrainer(
    model=model,
    train_dataset=dataset["train"],
    args=SFTConfig(output_dir="./resultats"),
)
trainer.train()
Décorticage
SFTTrainer"Supervised Fine-Tuning" — une version du Trainer standard, pré-adaptée aux formats instruction/réponse
SFTConfigvariante de TrainingArguments (vu au Module 04) avec des réglages par défaut pensés pour les LLM
Le reste du cours utilise le Trainer générique de transformers — plus verbeux mais plus universel, et donc plus formateur pour comprendre ce qui se passe réellement.
Module 03

Préparer les données

La qualité du dataset détermine la qualité du modèle fine-tuné

03.1 — La base de tout

Format des données d'entraînement

Pour un fine-tuning par instructions, chaque exemple d'entraînement est une paire "ce qu'on demande au modèle" / "ce qu'il devrait répondre". Le format le plus courant est le JSON Lines (un objet JSON par ligne).

// fichier mon_dataset.jsonl — un exemple par ligne
{"instruction": "Résume ce texte en une phrase.", "input": "...", "output": "..."}
{"instruction": "Classe ce ticket support.", "input": "...", "output": "Facturation"}
Décorticage
instructionla consigne donnée au modèle, constante ou variable selon les exemples
inputle contexte ou contenu sur lequel appliquer l'instruction (peut être vide pour certaines tâches)
outputla réponse EXACTE que le modèle doit apprendre à produire — la qualité de ce champ est ce qui compte le plus
Règle d'or : 200 exemples de très haute qualité, cohérents et représentatifs, valent souvent mieux que 10 000 exemples bruités ou incohérents entre eux.
03.2 — En pratique

Charger et explorer un dataset

Une fois le dataset chargé (via le Hub ou un fichier local), on l'explore toujours avant de l'utiliser — pour repérer les erreurs de format ou de contenu tôt.

dataset = load_dataset("json", data_files="mon_dataset.jsonl", split="train")

print(len(dataset))       # nombre d'exemples
print(dataset[0])       # premier exemple, sous forme de dictionnaire
print(dataset.column_names)  # noms des champs disponibles
Décorticage
split="train"indique qu'on charge directement le sous-ensemble "train", pas un dictionnaire de sous-ensembles
dataset[0]accès par index, comme une liste — utile pour inspecter visuellement quelques exemples
03.3 — Séparer pour évaluer

Découper en jeu d'entraînement et de test

On garde toujours une partie des données à l'écart de l'entraînement, pour vérifier ensuite que le modèle généralise bien — et ne se contente pas de "mémoriser" ce qu'il a vu.

dataset_split = dataset.train_test_split(test_size=0.1, seed=42)

train_dataset = dataset_split["train"]  # 90% des exemples
eval_dataset = dataset_split["test"]    # 10% des exemples, jamais vus à l'entraînement
Décorticage
test_size=0.1réserve 10% des exemples pour l'évaluation (proportion courante : 5 à 20%)
seed=42fige le hasard du découpage : relancer le code produit exactement le même split, pour des résultats reproductibles
03.4 — Rendre le texte compréhensible pour le modèle

Tokeniser le dataset

Un modèle ne "lit" pas du texte brut : il manipule des nombres (tokens). Il faut donc transformer chaque exemple texte en identifiants numériques, via le tokenizer associé au modèle.

def tokenize(exemple):
    return tokenizer(exemple["input"], padding="max_length", truncation=True)

dataset_tokenise = dataset.map(tokenize, batched=True)
Décorticage
padding="max_length"complète les textes courts pour que tous les exemples d'un lot aient la même longueur (nécessaire pour l'entraînement en lots)
truncation=Truecoupe les textes trop longs à la limite maximale du modèle, plutôt que de provoquer une erreur
.map(fn, batched=True)applique la fonction à TOUT le dataset, par lots (plus rapide qu'exemple par exemple)
Module 04

Fine-tuning pas à pas

Du modèle pré-entraîné au modèle spécialisé, avec le Trainer

04.1 — Le point de départ

Charger le modèle pré-entraîné

On charge un modèle déjà entraîné sur du texte général, qui servira de socle. Le choix de la classe "AutoModelFor..." dépend de la tâche visée.

from transformers import AutoModelForCausalLM

model_name = "mistralai/Mistral-7B-v0.1"
model = AutoModelForCausalLM.from_pretrained(model_name)
Décorticage
AutoModelForCausalLMarchitecture pour la GÉNÉRATION de texte (prédire le mot suivant) — utilisée pour les chatbots/assistants
"mistralai/Mistral-7B-v0.1"identifiant Hub du modèle : organisation "mistralai", modèle "Mistral-7B", version "v0.1"
🧪 Mini-exemple — une autre tâche, une autre classe
from transformers import AutoModelForSequenceClassification
model = AutoModelForSequenceClassification.from_pretrained("distilbert-base-uncased", num_labels=3)
Pour une tâche de CLASSIFICATION (ex : sentiment positif/neutre/négatif), on utilise AutoModelForSequenceClassification à la place.
04.2 — Régler l'entraînement

TrainingArguments

Cet objet centralise tous les réglages de l'entraînement (durée, vitesse d'apprentissage, fréquence de sauvegarde...). C'est ici que se jouent la plupart des ajustements d'un fine-tuning.

from transformers import TrainingArguments

training_args = TrainingArguments(
    output_dir="./resultats",
    num_train_epochs=3,
    per_device_train_batch_size=8,
    learning_rate=2e-5,
    eval_strategy="epoch",
    save_strategy="epoch",
)
Décorticage des hyperparamètres clés
num_train_epochscombien de fois l'entraînement parcourt l'intégralité du dataset (souvent 2 à 5 pour du fine-tuning)
batch_sizenombre d'exemples traités simultanément — limité par la mémoire GPU disponible
learning_ratel'ampleur de l'ajustement des poids à chaque étape — trop élevé casse le modèle, trop bas rend l'entraînement inefficace
eval_strategy"epoch" = évalue le modèle sur eval_dataset après chaque passage complet du dataset
04.3 — Lancer l'entraînement

Trainer et trainer.train()

Le Trainer assemble tout ce qu'on a préparé (modèle, données, réglages) et gère la boucle d'entraînement complète — calcul de l'erreur, ajustement des poids, sauvegardes.

from transformers import Trainer

trainer = Trainer(
    model=model,
    args=training_args,
    train_dataset=train_dataset,
    eval_dataset=eval_dataset,
)

trainer.train()
Décorticage
Trainer(...)assemble les quatre ingrédients : le modèle à entraîner, ses réglages, et les deux jeux de données préparés aux Modules 03 et 04
trainer.train()démarre réellement l'entraînement — peut durer de quelques minutes à plusieurs heures selon la taille du modèle et des données
Après l'entraînement, on sauvegarde le résultat avec trainer.save_model("./mon-modele") — le Module 07 détaille comment le partager sur le Hub.
04.4 — Vérifier que ça se passe bien

Suivre l'entraînement en direct

Pendant trainer.train(), la console affiche la loss (erreur du modèle) à intervalles réguliers — la métrique la plus importante à surveiller en direct.

# Extrait de la console pendant l'entraînement
{'loss': 2.14, 'epoch': 0.5}
{'loss': 1.62, 'epoch': 1.0}
{'eval_loss': 1.58, 'epoch': 1.0}
Comment interpréter
loss ↓doit globalement DIMINUER au fil de l'entraînement : le modèle apprend
eval_losscalculée sur les données JAMAIS vues à l'entraînement — la seule qui indique si le modèle généralise vraiment
Signal d'alerte : si la loss d'entraînement continue de baisser mais que l'eval_loss remonte, c'est le signe d'un surapprentissage (overfitting) — voir Module 06.
Module 05

LoRA / QLoRA en détail

Fine-tuner un modèle de plusieurs milliards de paramètres sur une seule carte graphique

05.1 — Le problème à résoudre

Pourquoi le full fine-tuning est coûteux

Réentraîner TOUS les paramètres d'un grand modèle demande de stocker en mémoire GPU : les poids du modèle, les gradients (l'ajustement calculé pour chaque poids), et l'état de l'optimiseur — plusieurs fois la taille du modèle.

  • Un modèle de 7 milliards de paramètres en full fine-tuning peut nécessiter plus de 80 Go de mémoire GPU
  • Ce niveau de mémoire n'est disponible que sur du matériel professionnel coûteux
  • D'où l'intérêt de PEFT : obtenir un résultat proche, avec une fraction de cette mémoire
05.2 — L'idée clé

Le principe de LoRA

LoRA (Low-Rank Adaptation) part d'une observation : l'AJUSTEMENT nécessaire pour spécialiser un modèle peut être représenté par des matrices beaucoup plus petites que le modèle lui-même, sans perte de qualité notable.

  • Les poids ORIGINAUX du modèle restent gelés (non modifiés)
  • On ajoute, à côté, deux petites matrices "A" et "B" — les seules à être entraînées
  • Le résultat final combine : sortie du modèle gelé + ajustement des petites matrices
  • Concrètement, on entraîne souvent moins de 1% du nombre total de paramètres
Analogie : plutôt que de repeindre entièrement une maison (full fine-tuning), LoRA ajoute une fine couche de vernis ciblée qui change l'aspect final, sans toucher à la structure existante.
05.3 — En pratique

Configurer LoRA avec peft

Trois lignes suffisent pour transformer un modèle standard en modèle "LoRA-isé" : on définit la configuration, puis on l'applique au modèle chargé au Module 04.

from peft import LoraConfig, get_peft_model, TaskType

lora_config = LoraConfig(
    task_type=TaskType.CAUSAL_LM,
    r=8,
    lora_alpha=16,
    lora_dropout=0.05,
    target_modules=["q_proj", "v_proj"],
)

model = get_peft_model(model, lora_config)
model.print_trainable_parameters()
Décorticage des paramètres LoRA
r=8le "rang" des petites matrices ajoutées — plus il est élevé, plus le modèle peut apprendre de nuances, mais plus l'entraînement coûte cher (valeurs courantes : 4 à 64)
lora_alphaun facteur d'échelle qui pondère l'influence des matrices LoRA sur la sortie finale
target_modulesles couches précises du modèle où appliquer LoRA — souvent les couches d'attention ("query" et "value")
05.4 — Aller encore plus loin

QLoRA : LoRA + quantification

QLoRA ajoute une étape supplémentaire : le modèle de base est chargé en précision réduite (4 bits au lieu de 16 ou 32 bits), ce qui divise sa taille en mémoire par 4 à 8, avant d'y appliquer LoRA.

from transformers import BitsAndBytesConfig

bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",
    bnb_4bit_compute_dtype="bfloat16",
)

model = AutoModelForCausalLM.from_pretrained(model_name, quantization_config=bnb_config)
Décorticage
load_in_4bitcharge les poids du modèle en 4 bits plutôt qu'en 16/32 bits — moins précis, mais beaucoup plus léger en mémoire
bnb_4bit_quant_type"nf4" (NormalFloat4) est un format de quantification optimisé, qui préserve mieux la qualité que d'autres méthodes 4 bits
QLoRA est ce qui permet de fine-tuner des modèles de 7 à 13 milliards de paramètres sur une seule carte grand public (ex : 24 Go de VRAM), là où le full fine-tuning serait impossible.
05.5 — Vérifier le gain

Mesurer ce qu'on entraîne vraiment

La méthode print_trainable_parameters() (utilisée à la fin de la slide 05.3) affiche concrètement le gain apporté par LoRA — un chiffre qui parle souvent plus que n'importe quelle explication théorique.

🧪 Mini-exemple — ce que la console affiche
trainable params: 4,194,304 || all params: 7,245,533,184 || trainable%: 0.0579
Ici, seulement 0,058% des 7,2 milliards de paramètres du modèle sont réellement entraînés — le reste (99,94%) reste gelé, tel quel.
C'est ce chiffre qui explique pourquoi LoRA rend le fine-tuning accessible : on ajuste quelques millions de paramètres au lieu de plusieurs milliards.
Module 06

Évaluer le modèle fine-tuné

S'assurer que le fine-tuning a vraiment amélioré ce qu'on visait

06.1 — Le test le plus simple

Comparer le modèle avant et après

Avant toute métrique chiffrée, la première vérification consiste à comparer les réponses du modèle ORIGINAL et du modèle FINE-TUNÉ sur les mêmes exemples.

from transformers import pipeline

original = pipeline("text-generation", model="mistralai/Mistral-7B-v0.1")
finetune  = pipeline("text-generation", model="./mon-modele-finetune")

prompt = "Rédige une réponse de support pour un client insatisfait."
print("AVANT :", original(prompt)[0]["generated_text"])
print("APRÈS :", finetune(prompt)[0]["generated_text"])
Décorticage
pipeline("text-generation", ...)crée un objet prêt à l'emploi qui encapsule modèle + tokenizer + logique de génération
model="./mon-modele-finetune"un chemin local fonctionne exactement comme un identifiant du Hub
06.2 — Chiffrer la qualité

Métriques d'évaluation

Selon la tâche, différentes métriques permettent de mesurer objectivement la qualité du modèle fine-tuné, au-delà de l'observation manuelle.

  • Perplexité : mesure à quel point le modèle est "surpris" par un texte — plus basse = meilleure prédiction du langage
  • Accuracy / F1-score : pour les tâches de classification, comparaison directe avec les vraies réponses attendues
  • Évaluation humaine : indispensable pour les tâches génératives (résumé, rédaction) où plusieurs réponses correctes existent
  • eval_loss (vu au Module 04) : suivi en continu pendant l'entraînement, sur les données jamais vues
Aucune métrique automatique ne remplace totalement une relecture humaine d'un échantillon de sorties — surtout pour les tâches génératives ouvertes.
06.3 — Ce qui peut mal tourner

Risques à surveiller

  • Overfitting (surapprentissage) : le modèle "récite" les exemples d'entraînement au lieu de généraliser — détecté quand l'eval_loss remonte alors que la loss d'entraînement continue de baisser
  • Catastrophic forgetting : le modèle devient très bon sur la nouvelle tâche mais perd des capacités générales qu'il avait avant le fine-tuning
  • Biais amplifiés : un dataset de fine-tuning déséquilibré peut renforcer des biais déjà présents, ou en introduire de nouveaux
Parades courantes : garder un jeu d'évaluation représentatif, limiter le nombre d'epochs, et tester le modèle final sur des exemples "hors sujet" pour vérifier qu'il n'a rien oublié d'important.
Module 07

Déployer & partager

Du modèle entraîné localement au modèle utilisable en production

07.1 — Publier son travail

push_to_hub()

Une fois satisfait du résultat, on peut publier le modèle fine-tuné (et son tokenizer) directement sur le Hub — privé ou public — en une ligne de code.

model.push_to_hub("mon-compte/mon-modele-finetune")
tokenizer.push_to_hub("mon-compte/mon-modele-finetune")
Décorticage
push_to_hub(id)crée automatiquement le dépôt sur le Hub s'il n'existe pas encore, puis y téléverse les fichiers
"mon-compte/..."requiert d'être authentifié au préalable (huggingface-cli login) avec un compte Hugging Face
Publier son tokenizer en même temps que le modèle est essentiel : sans lui, personne ne peut réutiliser correctement le modèle.
07.2 — Documenter son modèle

Rédiger une Model Card

Chaque dépôt de modèle sur le Hub a un fichier README.md appelé "Model Card" — la documentation qui permet à d'autres (ou à vous-même dans 6 mois) de comprendre et réutiliser le modèle correctement.

  • Description : quel modèle de base, quelle tâche, quel objectif
  • Données d'entraînement : source, taille, période, éventuelles limitations
  • Comment l'utiliser : exemple de code minimal fonctionnel
  • Limites connues : cas où le modèle performe mal, biais identifiés
  • Métriques d'évaluation : les résultats chiffrés du Module 06
Une Model Card incomplète est l'une des premières causes de mauvaise réutilisation d'un modèle par une autre équipe — elle fait partie intégrante du livrable, pas une option.
07.3 — Utiliser le modèle publié

Exercices sur Hugging Face

⚙️ Exercice pratique 1

Explorer le Hub pour un cas d'usage donné

15 min · Individuel · huggingface.co
  1. Cas d'usage : une entreprise veut classer automatiquement des tickets support en français par catégorie
  2. Cherchez sur le Hub un modèle de base pertinent (filtrez par tâche "Text Classification" et langue "French")
  3. Cherchez un dataset existant qui pourrait servir de point de départ ou d'inspiration de format
  4. Notez les identifiants trouvés (format organisation/nom) et justifiez vos choix en 2-3 phrases
Livrable attendu : un modèle et un dataset identifiés, avec une justification écrite du choix.
⚙️ Exercice pratique 2

Écrire une configuration LoRA

15 min · Individuel · À partir du Module 05
  1. Scénario : vous disposez d'un seul GPU avec 16 Go de VRAM pour fine-tuner un modèle de 7 milliards de paramètres
  2. Écrivez un bloc LoraConfig complet et cohérent avec cette contrainte de mémoire
  3. Justifiez votre choix de r (rang) : pourquoi pas 64 ? pourquoi pas 1 ?
  4. Ajoutez un bloc BitsAndBytesConfig (QLoRA) pour tenir dans la contrainte mémoire
Livrable attendu : les deux blocs de configuration Python, commentés, avec la justification du rang choisi.
⚙️ Exercice pratique 3 — bonus

Rédiger une Model Card

10 min · Individuel · Sur le modèle imaginaire de l'exercice 1
  1. En reprenant le cas d'usage de classification de tickets support (exercice 1)
  2. Rédigez une Model Card minimale : description, données d'entraînement (imaginées), comment l'utiliser, limites connues
  3. Incluez un exemple de code d'utilisation avec pipeline()
Pourquoi cet exercice compte : un modèle non documenté, même excellent, est difficilement réutilisable par une équipe — la documentation fait partie du livrable technique, pas une formalité annexe.
Avant de conclure

Bonnes pratiques à retenir

  • Commencez toujours par vérifier si un prompt mieux conçu ou du RAG suffisent avant de fine-tuner
  • Privilégiez LoRA/QLoRA au full fine-tuning, sauf besoin très spécifique et gros moyens disponibles
  • Peu de données de qualité valent mieux que beaucoup de données bruitées
  • Gardez toujours un jeu d'évaluation séparé, jamais vu à l'entraînement
  • Documentez systématiquement via une Model Card avant de partager un modèle
  • Comparez avant/après sur des exemples concrets, pas seulement sur des métriques abstraites
