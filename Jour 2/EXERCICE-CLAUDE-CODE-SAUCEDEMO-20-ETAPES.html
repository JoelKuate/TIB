💡 Principe : de la user story au rapport de test, sans coder à la main

Vous ne décrivez plus une action technique ("clique ici, vérifie là") mais un besoin utilisateur. Claude Code traduit la user story en test Playwright, l'exécute, et consigne le résultat dans un rapport — automatiquement, à chaque fois.

✍️ Sans Claude Code (tout à la main)

1. Traduire la user story en cas de test
2. Écrire le script Playwright
3. Lancer, lire l'erreur, corriger
4. Noter le résultat dans un fichier à part
5. Répéter pour chacune des 20 user stories

📋 Instruction-cadre — à donner UNE SEULE FOIS, avant la user story n°1

Cette instruction met en place le mode de travail pour toute la session : elle évite de répéter "génère aussi le rapport" à chacune des 20 étapes.

Prompt à donner à Claude Code (une seule fois)

Nous allons tester le site https://www.saucedemo.com à partir d'une série de user stories que je vais te donner une par une. Pour CHAQUE user story que je te transmets : 1. Groupe A — Authentification

01
NavigationConnexion réussie avec un utilisateur standard
User Story
En tant que client inscrit, je veux me connecter avec des identifiants valides, afin de pouvoir accéder à mon catalogue de produits.
Prompt / critère d'acceptation

Écris un test Playwright qui va sur https://www.saucedemo.com, se connecte avec l'utilisateur standard_user / secret_sauce, puis vérifie que l'URL contient bien "inventory.html" et que la liste de produits est visible.

Ce que Claude Code va générer
test('Cas 01 - connexion réussie avec un utilisateur standard', async ({ page }) => {
  await page.goto('https://www.saucedemo.com/');
  await page.getByTestId('username').fill('standard_user');
  await page.getByTestId('password').fill('secret_sauce');
  await page.getByTestId('login-button').click();
  await expect(page).toHaveURL(/inventory\.html/);
  await expect(page.getByTestId('inventory-list')).toBeVisible();
});
✅ Résultat attendu : redirection vers inventory.html, la liste des 6 produits est visible.

07
AssertionLe tri par prix croissant réordonne réellement
User Story
En tant que client, je veux trier les produits du moins cher au plus cher, afin de trouver rapidement les meilleures affaires.
Prompt / critère d'acceptation

Sélectionne le tri "Price (low to high)" dans le menu déroulant du catalogue. Récupère tous les prix affichés, et vérifie par le calcul (pas juste visuellement) qu'ils sont bien classés du plus petit au plus grand.

Ce que Claude Code va générer

const prixTextes = await page.getByTestId('inventory-item-price').allTextContents();
const prix = prixTextes.map(t => parseFloat(t.replace('$', '')));
const prixTries = [...prix].sort((a, b) => a - b);

expect(prix).toEqual(prixTries);
✅ Résultat attendu : la liste des prix affichés est identique à sa version triée — le tri fonctionne vraiment.
💡 Conseil formateur : c'est la différence entre "l'interface dit qu'elle a trié" et "l'interface a vraiment trié" — un piège classique des tests trop superficiels.

Groupe F — Cas avancé : régression visuelle

Cas 20

Un cas plus subtil qui documente un bug connu du site, sans casser le tableau vert de la suite.

20
AssertionDétecter que les images produits sont bien distinctes (et documenter le cas où elles ne le sont pas)
User Story
En tant que responsable qualité, je veux détecter si les images produits sont correctement affichées (et documenter les cas où ce n'est pas le cas), afin de garantir la fiabilité visuelle du catalogue.
Prompt / critère d'acceptation

Écris deux tests : (a) avec standard_user, vérifie que les 6 images produits du catalogue ont bien 6 sources différentes ; (b) avec problem_user (compte volontairement buggé par le site), documente que ce même contrôle échoue — sans faire échouer toute la suite.

Ce que Claude Code va générer
test('Cas 20a - standard_user affiche 6 images distinctes', async ({ page }) => {
  // ... connexion standard_user ...
  const sources = await page.locator('.inventory_item_img img').evaluateAll(
    imgs => imgs.map(i => (i as HTMLImageElement).src)
  );
  expect(new Set(sources).size).toBe(6);
});

test('Cas 20b - bug connu : problem_user affiche 6 fois la même image', async ({ page }) => {
  test.fail(); // documente un défaut connu, sans casser la suite
  // ... connexion problem_user ...
  const sources = await page.locator('.inventory_item_img img').evaluateAll(
    imgs => imgs.map(i => (i as HTMLImageElement).src)
  );
  expect(new Set(sources).size).toBe(6);
});
✅ Résultat attendu : (a) 6 images distinctes pour standard_user · (b) échec confirmé et volontairement documenté pour problem_user.
💡 Conseil formateur : test.fail() est l'outil idéal pour documenter un bug connu : le test reste dans la suite (traçabilité), n'est pas compté comme un échec inattendu, et alertera automatiquement si le bug venait à être corrigé sans mise à jour du test.

📊 Étape finale — transformer le rapport brut en synthèse professionnelle

Une fois les 20 user stories testées, demandez à Claude Code de mettre en forme le résultat pour une présentation à l'équipe ou au client.

Le fichier RAPPORT-TESTS-SAUCEDEMO.md contient maintenant les résultats des 20 user stories testées.

Transforme-le en un rapport de synthèse :

1. Résumé en tête — nombre de user stories testées, taux de réussite, liste des cas en échec (attendus ou non).

2. Résultats groupés par thème — Authentification, Catalogue, Panier, Fiche produit, Checkout, Cas avancé — reprends pour chaque cas la user story, le statut et l'observation.

3. Fiches défauts — pour chaque ❌, précise si le défaut est connu et documenté (comme le cas 20b) ou inattendu (à investiguer).

4. Recommandation finale — conclus par un avis GO / NO-GO pour la mise en production, justifié en une phrase.
